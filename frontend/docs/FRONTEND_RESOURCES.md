**Version:** 1.0

**Status:** Active Development

**Phase:** Frontend Module 1

**Last Updated:** June 2026

---

# 1. Purpose

This document serves as the **single source of truth** for frontend development throughout the entire InfoLand AI project.

Its purpose is to ensure that every frontend implementation:

* follows a consistent architecture
* integrates correctly with the backend
* never assumes backend fields
* remains modular and maintainable
* is reusable across future modules

This document should be referenced before implementing any frontend feature.

---

# 2. Project Overview

## Project Name

InfoLand AI

## Primary Goal

Build an AI-powered Property Verification and Risk Assessment Platform.

The application is **NOT** a property listing website.

Its purpose is to answer:

> Is this property legally, financially and structurally safe to purchase?

The frontend must visualize and present information generated by the backend.

Business logic must remain inside backend services.

# 6. Frontend Development Principles

Every implementation must follow these principles.

## Principle 1

Backend is the Single Source of Truth.

Frontend must never invent fields.

---

## Principle 2

Never hardcode backend responses.

Read actual API responses.

---

## Principle 3

Never assume endpoint names.

Always use implemented backend routes.

---

## Principle 4

Business logic belongs inside Services.

Components render UI only.

---

## Principle 5

Redux manages shared application state.

Local state manages UI state only.

---

## Principle 6

Prefer reusable components over duplicated UI.

---

## Principle 7

Implement features one module at a time.

---

# 9. Component Guidelines

Every component must

* use Functional Components
* use Hooks
* have one component per file
* use PascalCase
* export default
* remain focused on presentation

Components should never communicate directly with backend APIs.

---

# 10. Service Layer Rules

Only Services communicate with backend.

Flow

```text
React Component

↓

Redux AsyncThunk

↓

PropertyService

↓

Axios Client

↓

Backend
```

Never call Axios directly from pages or components.

# 14. Field Usage Rules

Frontend should only use backend fields.

Never create frontend-only business fields.

Examples

Correct

```text
property_id

property_class

sale_status

geometry

centroid

location_score
```

Incorrect

```text
plotSafety

ownerScore

verificationPercent

riskLabel
```

Unless returned by backend.

# 21. Golden Rules

✓ Backend is the source of truth.

✓ Never invent endpoints.

✓ Never invent response fields.

✓ Never modify backend contracts.

✓ Read Swagger before implementation.

✓ Read backend module documentation before coding.

✓ Keep components reusable.

✓ Keep services independent.

✓ Keep Redux predictable.

✓ Test every feature.

✓ Commit after every completed task.

✓ Remove dummy JSON only after successful backend integration.
